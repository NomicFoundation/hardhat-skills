# TypeChain Migration Reference

This reference covers ensuring all TypeChain-generated types compile correctly after migrating to Hardhat V3 + ESM. The goal: every file that imports from the TypeChain output directory must have zero import errors — no `// @ts-ignore`, no `any` casts, no missing types.

---

## Rules

- **No casts allowed.** Do not add `as any`, `// @ts-ignore`, or `// @ts-expect-error` to work around missing types. Fix the root cause.
- **No manual files.** Do not create or edit files inside the TypeChain output folder. All files there are auto-generated by `npx hardhat build`. If types are missing, fix the config so the build generates them.
- **Single centralized folder.** All TypeChain types must live in one folder. If the V2 project already used TypeChain, keep the same output folder name (e.g., `typechain-types` or `types`). Set `typechain.outDir` in `hardhat.config.ts` to match.

---

## Prerequisite: the TypeChain plugin

TypeChain generation is **not** in Hardhat 3 core. `npx hardhat build` only emits types when the `@nomicfoundation/hardhat-typechain` plugin is registered — it is bundled in `@nomicfoundation/hardhat-toolbox-mocha-ethers` (the legacy `@typechain/hardhat` plugin does not support Hardhat 3, and the viem toolbox uses `@nomicfoundation/hardhat-viem` instead). If `build` produces no types, confirm the plugin is installed and listed in the config `plugins` array before diagnosing further.

## Initial build

Before entering the per-file loop, run `npx hardhat build` once to compile contracts and generate TypeChain types. If the build fails or types are missing, work through the diagnosis checklist at the bottom before continuing.

---

## Per-file migration loop (strictly sequential)

Process files **one at a time**. For each file that imports from the TypeChain output directory, execute **all three steps below in order**. **Do NOT move to the next file until step 3 confirms zero compilation errors for the current file.**

### Step 1 — Rebuild types (only if config changed)

If fixing the current file required adding new entries to `paths.sources.solidity` or `solidity.npmFilesToBuild` in `hardhat.config.ts`, run `npx hardhat build --force` to regenerate types for the newly added sources. If no config change was needed for this file, skip this step.

### Step 2 — Fix imports

Under ESM, bare directory imports no longer auto-resolve to `index.js`. Update the imports in the current file to use explicit `.js` extensions.

```ts
// V2 — bare directory import (CJS auto-resolves to index)
import {
  LidoLocator,
  LidoLocator__factory,
  OssifiableProxy,
  OssifiableProxy__factory,
} from "typechain-types";

// V3 — explicit index.js (ESM requires it)
import {
  LidoLocator,
  LidoLocator__factory,
  OssifiableProxy,
  OssifiableProxy__factory,
} from "typechain-types/index.js";
```

This applies to **all** imports from the TypeChain output directory:

- Barrel imports: `"typechain-types"` → `"typechain-types/index.js"`
- Type imports: `import type { Dashboard } from "typechain-types"` → `from "typechain-types/index.js"`
- Factory imports: `"typechain-types/factories/MyContract__factory"` → `"typechain-types/factories/MyContract__factory.js"`
- Deep imports: any sub-path also needs `.js` on the final segment

If the project uses a different `typechain.outDir` (e.g., `types`), replace `typechain-types` with that name throughout.

> **V3's output layout may differ from V2's — it's not always just `.js` appending.** TypeChain strips the **lowest common path** of all built artifacts, so the output tree depends on how many source roots the build has — this is not a fixed V2-vs-V3 rule. With a **single source root** (only `contracts/` is built), that `contracts/` prefix is the common path and **disappears** from the output: a contract at `contracts/factories/Foo.sol` lands at `typechain-types/factories/Foo`, and its factory at `typechain-types/factories/Foo__factory` — so source-contract types and `__factory` classes can share one `factories/` dir. With **multiple roots** (e.g. external `@openzeppelin` artifacts, or a non-empty `npmFilesToBuild`), the common prefix is higher up and `contracts/` is **preserved** (e.g. `typechain-types/contracts/acl/ACL` — see §"Fixing imports not re-exported" below). So a V2 import path containing `/contracts/` may need that segment **removed** (single-root) or may keep it (multi-root) — and either way may need `.js` added. Always `ls` the regenerated tree to confirm the real path before rewriting.

#### Fixing imports not re-exported from `index.ts`

Some types exist as individual files in the TypeChain output directory but are **not** re-exported from `index.ts`. When this happens, `npx tsc` will report an error like `Module '"typechain-types/index.js"' has no exported member 'ACL'`.

**Do not edit `index.ts`** — it is auto-generated. Instead, split the import: move the missing types to direct file imports and keep the rest on the barrel import.

```ts
// BEFORE — ACL is not re-exported from index.ts, so this fails:
import {
  ACL,
  Lido,
  LidoLocator,
  StakingRouter__MockForLidoMisc,
  WithdrawalQueue__MockForLidoMisc,
} from "typechain-types/index.js";

// AFTER — import ACL directly from its own file:
import { ACL } from "typechain-types/ACL.js";
import {
  Lido,
  LidoLocator,
  StakingRouter__MockForLidoMisc,
  WithdrawalQueue__MockForLidoMisc,
} from "typechain-types/index.js";
```

**How to identify the correct file:** TypeChain mirrors the contract's source directory layout, so a type's file is not always at the output root. If `SomeType` is missing from `index.ts`, use `ls`/`find` in the TypeChain output directory to locate its actual file — it may be nested (e.g. `typechain-types/contracts/acl/ACL.js`), not just `typechain-types/ACL.js`. Import from wherever it actually lives, with a `.js` extension (ESM resolution).

### Step 3 — Verify (gate — must pass before next file)

Verify the file you just edited compiles, filtering `tsc` output to it:

```bash
npx tsc --noEmit --pretty 2>&1 | grep {filename}
```

A TypeChain import error is reported against the importing file, so this catches it — no scratch file or `tsconfig` edit is needed. (`tsconfig.json` is owned by the `tsconfig` domain; do not edit it here.)

**Zero output = pass.** If there is any output, the imports have errors:

- **Missing type name** (e.g. `Module '"typechain-types/index.js"' has no exported member 'ACL'`) → the contract is not being compiled. Go to the diagnosis checklist below, then return to **Step 1** for this same file.
- **Cannot find module** → the import path is wrong. Fix the `.js` extension or `typechain.outDir` mismatch in **Step 2**, then re-run this verification.
- **Repeat steps 1–3 for this file** until there is **zero** grep output.

**Only after zero output:** move to the next file.

## Diagnosis checklist — when types are missing

Work through in order:

1. **Missing types for some contracts?** → `paths.sources.solidity` in `hardhat.config.ts` probably only includes `"contracts"`. Hardhat only compiles — and therefore TypeChain only generates types for — directories listed in that array. Search the entire project for `.sol` and `.t.sol` files to find all directories containing Solidity sources (e.g., `test/`, `lib/`, `src/`).
   - **Action:** In the `config` domain, add the discovered directories to `paths.sources.solidity` (in sub-agent mode, hand this off to the config domain rather than editing the config yourself). Then run `npx hardhat build --force` and verify the missing types are now generated.

2. **Missing types for npm contracts?** → The contracts are not listed in `solidity.npmFilesToBuild`. This is common for OpenZeppelin and other npm-installed contract libraries. Examples:

   ```ts
   solidity: {
     npmFilesToBuild: [
       "@openzeppelin/contracts/token/ERC20/IERC20.sol",
       "@openzeppelin/contracts/token/ERC721/ERC721.sol",
     ],
   }
   ```

   - **Action:** In the `config` domain, add the specific npm contract paths to `solidity.npmFilesToBuild` (hand off in sub-agent mode). Then run `npx hardhat build --force` and verify the missing types are now generated.

3. **Types exist on disk but imports fail?** → `typechain.outDir` in `hardhat.config.ts` doesn't match the actual folder TypeChain generated. Run `ls` on the project root after `npx hardhat build` to find the real folder name (e.g., `types` vs `typechain-types`).
   - **Action:** In the `config` domain, update `typechain.outDir` to the actual folder name (hand off in sub-agent mode).

After each config fix, always run `npx hardhat build --force` to regenerate types, then re-verify the failing imports.
